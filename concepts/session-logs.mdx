spec-loop creates comprehensive logs for every run, providing both human-readable summaries and machine-readable analytics for debugging, cost tracking, and performance monitoring.

## Session Directory Structure

Each run creates a timestamped session directory:

```bash
.spec-loop/sessions/20260301_143022_user-auth/
├── session.json              # Machine-readable analytics
├── session.md                # Human-readable iteration summaries
├── run.md                    # Full run log with prompts and outputs
└── .circuit_breaker.json     # Circuit breaker state (internal)
```

<Note>
Session directories are named: `YYYYMMDD_HHMMSS_<spec-name>`, making them sortable by time.
</Note>

### Key Fields

<Accordion title="Session Metadata">
  | Field | Type | Description |
  |-------|------|-------------|
  | `session_id` | string | Unique identifier (same as directory name) |
  | `version` | string | spec-loop version that created this session |
  | `spec` | string | Path to spec directory |
  | `spec_name` | string | Short name extracted from spec path |
  | `started_at` | ISO 8601 | Session start timestamp |
  | `ended_at` | ISO 8601 | Session end timestamp (null if running) |
  | `duration_seconds` | number | Total session duration |
  | `exit_reason` | string | Why session ended (see table below) |
</Accordion>

<Accordion title="Exit Reasons">
  | Exit Reason | Meaning |
  |-------------|----------|
  | `ALL_TASKS_COMPLETE` | All tasks marked `done` |
  | `BLOCKED` | Task cannot proceed, needs human help |
  | `NO_PENDING_TASKS` | No tasks with status `pending` (may have `blocked` tasks) |
  | `ONCE` | User ran with `--once` flag |
  | `TASK_LIMIT` | Reached `--max-tasks` limit |
  | `MAX_ITERATIONS` | Reached `--max-loops` limit |
  | `CIRCUIT_OPEN` | Circuit breaker tripped |
  | `USER_INTERRUPT` | User pressed Ctrl+C |
</Accordion>

<Accordion title="Cost and Telemetry">
  | Field | Type | Description |
  |-------|------|-------------|
  | `total_cost_usd` | number | Total Claude API cost for session |
  | `total_iterations` | number | Number of build-review-fix cycles |
  | `max_loops` | number | Configured iteration limit |
  | `max_review_fix_loops` | number | Configured review-fix limit per task |
  | `claude_sessions` | string[] | All Claude session IDs used |
</Accordion>

<Accordion title="Invocations">
  Tracks each `spec-loop run` command within the session (sessions can span multiple invocations via `--once` or `--max-tasks`).

  | Field | Type | Description |
  |-------|------|-------------|
  | `started_at` | ISO 8601 | When this invocation started |
  | `mode` | string | Command mode (usually "run") |
  | `once` | boolean | `--once` flag was used |
  | `max_tasks` | number | `--max-tasks` value (0 = unlimited) |
  | `skip_review` | boolean | `--skip-review` flag was used |
  | `claude_session_ids` | string[] | Claude sessions for this invocation |
</Accordion>

<Accordion title="Iterations">
  Per-iteration metrics for each task attempt.

  | Field | Type | Description |
  |-------|------|-------------|
  | `index` | number | Iteration number (1-indexed) |
  | `task` | string | Task name being implemented |
  | `outcome` | string | `PASS`, `FAIL`, `BLOCKED` |
  | `duration_seconds` | number | Time for build+review+fix cycle |
  | `cost_usd` | number | Claude API cost for this iteration |
  | `must_fix_count` | number | Must-fix issues from review |
  | `should_fix_count` | number | Should-fix issues from review |
  | `commit` | string | Git commit SHA (short) |
  | `timestamp` | ISO 8601 | When iteration completed |
</Accordion>

## Session Creation and Continuation

### Initial Session

Created by `ensure_session_initialized()` (src/session.rs:140-189):

```rust
pub fn ensure_session_initialized(
    session_path: &Path,
    spec_dir: &Path,
    spec_name: &str,
    cfg: &Config,
    version: &str,
) -> Result<()> {
    fs::create_dir_all(session_path)?;

    let json_path = session_json_path(session_path);
    if !json_path.exists() {
        let data = SessionJson {
            session_id: session_path.file_name()
                .map(|s| s.to_string_lossy().to_string())
                .unwrap_or_else(|| "unknown-session".to_string()),
            version: version.to_string(),
            spec: spec_dir.display().to_string(),
            spec_name: spec_name.to_string(),
            started_at: now_iso(),
            max_loops: cfg.max_loops,
            max_review_fix_loops: cfg.max_review_fix_loops,
            max_tasks_per_run: cfg.max_tasks_per_run,
            claude_sessions: vec![],
            invocations: vec![],
            iterations: vec![],
            ended_at: None,
            duration_seconds: None,
            total_cost_usd: None,
            exit_reason: None,
            total_iterations: None,
        };
        write_session_json(&json_path, &data)?;
    }

    // Create run.md if it doesn't exist
    let run_path = session_runlog_path(session_path);
    if !run_path.exists() {
        let run_md = format!(
            "# Run Log\n\n- Spec: {}\n- Spec path: {}\n- Started: {}\n\n---\n",
            spec_name,
            spec_dir.display(),
            now_human(),
        );
        fs::write(&run_path, run_md)?;
    }

    Ok(())
}
```

### Session Continuation

spec-loop detects when to continue an existing session:

```rust
// src/session.rs:458-465
pub fn session_continuation_for_spec(session_dir: &Path, spec_dir: &Path) -> Option<PathBuf> {
    let latest = session_latest_for_spec(session_dir, spec_dir)?;
    let data = read_session_json(&session_json_path(&latest)).ok()?;
    match data.exit_reason.as_deref() {
        Some("ONCE") | Some("TASK_LIMIT") => Some(latest),
        _ => None,
    }
}
```

If the last session exited with `ONCE` or `TASK_LIMIT`, the next run continues the same session directory. This keeps all logs together for a complete feature implementation.

## Claude Session Tracking

spec-loop tracks Claude session IDs for debugging and audit purposes:

```rust
// src/session.rs:250-274
pub fn register_claude_session(session_path: &Path, claude_session_id: &str) -> Result<()> {
    let id = claude_session_id.trim();
    if id.is_empty() {
        return Ok(());
    }

    let json_path = session_json_path(session_path);
    if !json_path.exists() {
        return Ok(());
    }

    let mut data = read_session_json(&json_path)?;

    // Add to session-level claude_sessions array
    if !data.claude_sessions.iter().any(|v| v == id) {
        data.claude_sessions.push(id.to_string());
    }

    // Add to current invocation's claude_session_ids array
    if let Some(inv) = data.invocations.last_mut() {
        if !inv.claude_session_ids.iter().any(|v| v == id) {
            inv.claude_session_ids.push(id.to_string());
        }
    }

    write_session_json(&json_path, &data)
}
```

<Note>
Claude session IDs can be used to access detailed logs in Claude Code's session history.
</Note>

## Resume State

spec-loop stores resume state separately for `--resume` functionality:

```bash
.spec-loop/.session_state.json
```

Contents:
```json
{
  "spec_dir": ".agents/specs/user-auth",
  "loop_index": 3,
  "session_path": ".spec-loop/sessions/20260301_143022_user-auth",
  "phase": "review",
  "before_sha": "a3f91c2e7",
  "saved_at": "2026-03-01T14:35:45Z"
}
```

This allows resuming mid-iteration after interruptions.

<Warning>
Resume state is cleared automatically on successful run completion. It persists only for interrupted runs.
</Warning>

## Log Retention

Session logs are stored in `.spec-loop/sessions/` which should be **gitignored**:

```bash
# .gitignore
.spec-loop/
```

**Rationale:**
- Logs can be large (contain full prompts and outputs)
- Contain potentially sensitive data (API keys in outputs, code snippets)
- Are machine-specific (not useful in version control)

<Note>
If you need to share logs for debugging, manually copy specific session directories to a secure location.
</Note>
