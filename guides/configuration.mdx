spec-loop uses `.speclooprc` for project-specific configuration and environment variables for runtime overrides.

## Initialization

Generate `.speclooprc` with auto-detected settings:

```bash
spec-loop init
```

This creates:
- `.speclooprc` — Project configuration
- `.agents/` — Spec directories and templates
- `AGENTS.md` — Starter project rules file

<Tabs>
  <Tab title="Interactive (default)">
    ```bash
    spec-loop init
    ```
    
    Wizard prompts for:
    - Project type (auto-detected)
    - Verify command (auto-detected)
    - Test command (auto-detected)
  </Tab>
  
  <Tab title="Non-interactive">
    ```bash
    spec-loop init --no-wizard
    ```
    
    Uses auto-detected values without prompts.
  </Tab>
  
  <Tab title="Force overwrite">
    ```bash
    spec-loop init --force
    ```
    
    Overwrites existing `.speclooprc`.
  </Tab>
  
  <Tab title="Custom commands">
    ```bash
    spec-loop init \
      --verify-cmd "pnpm check" \
      --test-cmd "pnpm test"
    ```
    
    Specify commands directly.
  </Tab>
</Tabs>

## Configuration File (.speclooprc)

Generated by `spec-loop init`:

```bash
# spec-loop configuration
# Generated by spec-loop init

# Project
PROJECT_TYPE="typescript"
VERIFY_COMMAND="npm run lint && npm run typecheck"
TEST_COMMAND="npm test"

# Claude Code
# CLAUDE_MODEL=""

# Loop limits
# MAX_LOOPS=25
# MAX_REVIEW_FIX_LOOPS=3
# MAX_TASKS_PER_RUN=0

# Safety
# CB_NO_PROGRESS_THRESHOLD=3
# CB_COOLDOWN_MINUTES=30

# Paths
# SPECS_DIR=".agents/specs"
# SESSION_DIR=".spec-loop/sessions"
```

### Project Settings

<ParamField path="PROJECT_TYPE" type="string" default="(auto-detected)">
  Project type for documentation and context.
  
  Auto-detected types:
  - `typescript` — Has `tsconfig.json`
  - `javascript` — Has `package.json`
  - `python` — Has `pyproject.toml`, `setup.py`, or `setup.cfg`
  - `rust` — Has `Cargo.toml`
  - `go` — Has `go.mod`
  - `ruby` — Has `Gemfile`
  - `java` — Has `build.gradle` or `build.gradle.kts`
  - `swift` — Has `Package.swift`
  - `generic` — Fallback
</ParamField>

<ParamField path="VERIFY_COMMAND" type="string" default="(auto-detected)">
  Command to verify code quality (lint, typecheck, etc.).
  
  Runs after each build and during review. Build fails if this fails.
  
  Auto-detected commands:
  - TypeScript: `npm run lint && npm run typecheck`
  - JavaScript: `npm run lint`
  - Python: `python -m py_compile`
  - Rust: `cargo clippy && cargo check`
  - Go: `go vet ./...`
  - Ruby: `bundle exec rubocop`
  - Java: `./gradlew check`
  - Swift: `swift build`
</ParamField>

<ParamField path="TEST_COMMAND" type="string" default="(auto-detected)">
  Command to run tests.
  
  Runs after verify command during build and review.
  
  Auto-detected commands:
  - TypeScript/JavaScript: `npm test`
  - Python: `python -m pytest`
  - Rust: `cargo test`
  - Go: `go test ./...`
  - Ruby: `bundle exec rspec`
  - Java: `./gradlew test`
  - Swift: `swift test`
</ParamField>

<CodeGroup>
```bash TypeScript example
PROJECT_TYPE="typescript"
VERIFY_COMMAND="pnpm lint && pnpm typecheck"
TEST_COMMAND="pnpm test:unit"
```

```bash Python example
PROJECT_TYPE="python"
VERIFY_COMMAND="ruff check . && mypy ."
TEST_COMMAND="pytest tests/"
```

```bash Rust example
PROJECT_TYPE="rust"
VERIFY_COMMAND="cargo clippy -- -D warnings"
TEST_COMMAND="cargo test --all-features"
```

```bash Go example
PROJECT_TYPE="go"
VERIFY_COMMAND="go vet ./... && staticcheck ./..."
TEST_COMMAND="go test -race -cover ./..."
```
</CodeGroup>

### Claude Code Settings

<ParamField path="CLAUDE_MODEL" type="string" default="(Claude Code default)">
  Override Claude Code model.
  
  Leave empty to use Claude Code's default model selection.
  
  Example values:
  - `claude-opus-4-6` (most capable)
  - `claude-sonnet-4` (balanced)
  - `claude-haiku-4` (fastest, cheapest)
</ParamField>

<ParamField path="CLAUDE_BIN" type="string" default="claude" env="CLAUDE_BIN">
  Path to Claude Code binary.
  
  Override if `claude` is not in PATH or you have a custom installation.
</ParamField>

<ParamField path="CLAUDE_EXTRA_ARGS" type="string" default="" env="CLAUDE_EXTRA_ARGS">
  Additional arguments passed to `claude` command.
  
  Example:
  ```bash
  CLAUDE_EXTRA_ARGS="--timeout 300"
  ```
</ParamField>

### Loop Limits

<ParamField path="MAX_LOOPS" type="number" default="25" env="SPECLOOP_MAX_LOOPS">
  Maximum iterations before stopping.
  
  One iteration = build + review + fix (if needed).
  
  Set higher for complex features, lower for testing.
</ParamField>

<ParamField path="MAX_REVIEW_FIX_LOOPS" type="number" default="3" env="SPECLOOP_MAX_REVIEW_FIX_LOOPS">
  Review-fix retry attempts per task.
  
  If review still fails after this many attempts, task is left in `in-review` state.
</ParamField>

<ParamField path="MAX_TASKS_PER_RUN" type="number" default="0 (unlimited)" env="SPECLOOP_MAX_TASKS_PER_RUN">
  Maximum tasks to complete in one run.
  
  `0` = unlimited. Set to positive number to limit run duration.
</ParamField>

<CodeGroup>
```bash Extended run
# Allow longer runs for complex features
MAX_LOOPS=50
MAX_REVIEW_FIX_LOOPS=5
```

```bash Incremental work
# Complete 3 tasks per run
MAX_TASKS_PER_RUN=3
```

```bash Fast failure for testing
# Fail fast to test spec quality
MAX_LOOPS=5
MAX_REVIEW_FIX_LOOPS=1
```
</CodeGroup>

### Safety Settings

<ParamField path="CB_NO_PROGRESS_THRESHOLD" type="number" default="3">
  Circuit breaker: iterations without progress before triggering.
  
  Progress = tasks completed or status changed.
  
  Prevents infinite loops when model gets stuck.
</ParamField>

<ParamField path="CB_COOLDOWN_MINUTES" type="number" default="30">
  Circuit breaker cooldown period in minutes.
  
  After circuit opens, spec-loop won't run until cooldown expires.
</ParamField>

<Warning>
Do not set `CB_NO_PROGRESS_THRESHOLD` too high (>10) as it can lead to expensive loops without progress.
</Warning>

### Path Settings

<ParamField path="SPECS_DIR" type="string" default=".agents/specs" env="SPECLOOP_SPECS_DIR">
  Directory containing feature specs.
  
  Change if you prefer a different location.
</ParamField>

<ParamField path="SESSION_DIR" type="string" default=".spec-loop/sessions" env="SPECLOOP_SESSION_DIR">
  Directory for session logs.
  
  Should be gitignored (done automatically by `spec-loop init`).
</ParamField>

## Environment Variables

Environment variables override `.speclooprc` settings:

| Variable | Description | Example |
|----------|-------------|---------|
| `CLAUDE_BIN` | Claude Code binary path | `CLAUDE_BIN=/usr/local/bin/claude` |
| `CLAUDE_EXTRA_ARGS` | Extra args for claude | `CLAUDE_EXTRA_ARGS="--timeout 300"` |
| `SPECLOOP_MAX_LOOPS` | Override MAX_LOOPS | `SPECLOOP_MAX_LOOPS=50` |
| `SPECLOOP_MAX_REVIEW_FIX_LOOPS` | Override MAX_REVIEW_FIX_LOOPS | `SPECLOOP_MAX_REVIEW_FIX_LOOPS=5` |
| `SPECLOOP_MAX_TASKS_PER_RUN` | Override MAX_TASKS_PER_RUN | `SPECLOOP_MAX_TASKS_PER_RUN=10` |
| `SPECLOOP_SPECS_DIR` | Override SPECS_DIR | `SPECLOOP_SPECS_DIR=specs` |
| `SPECLOOP_SESSION_DIR` | Override SESSION_DIR | `SPECLOOP_SESSION_DIR=logs/sessions` |

<Tabs>
  <Tab title="One-off override">
    ```bash
    # Single run with custom limits
    SPECLOOP_MAX_LOOPS=50 spec-loop run
    ```
  </Tab>
  
  <Tab title="Session scope">
    ```bash
    # Set for current terminal session
    export SPECLOOP_MAX_TASKS_PER_RUN=5
    spec-loop run
    ```
  </Tab>
  
  <Tab title="Shell profile">
    ```bash
    # Add to ~/.bashrc or ~/.zshrc
    export CLAUDE_BIN="$HOME/bin/claude"
    export SPECLOOP_MAX_LOOPS=30
    ```
  </Tab>
</Tabs>

## CLI Arguments

CLI arguments have highest priority:

```bash
spec-loop run --max-loops 50 --max-tasks 10
```

**Priority order:**
1. CLI arguments (highest)
2. Environment variables
3. `.speclooprc` settings
4. Built-in defaults (lowest)

## Project Structure After Init

```text
your-project/
├── .speclooprc              # Project config
├── .agents/
│   ├── specs/               # Feature specs (created via /spec-loop-spec)
│   ├── templates/           # Spec/task templates (customizable)
│   │   ├── spec.md
│   │   ├── task.md
│   │   └── progress.md
│   ├── reference/           # Project patterns (you fill this)
│   └── decisions.md         # Design decisions log
├── .spec-loop/              # Runtime (gitignored)
│   └── sessions/            # Per-run logs
└── AGENTS.md                # Your project's rules
```

## AGENTS.md

The `AGENTS.md` file contains project conventions for Claude Code:

```text
# AGENTS.md

Project conventions and architecture rules for AI agents.

## Project Overview

[Describe what this project does, its architecture, and key technologies]

## Conventions

### File naming
- Components: PascalCase (UserProfile.tsx)
- Utilities: camelCase (formatDate.ts)
- Tests: *.test.ts

### Structure
- API routes in src/routes/
- Business logic in src/services/
- Data models in src/models/
- Shared utilities in src/utils/

### Patterns to follow
- Use async/await, not callbacks
- Validate input with Zod schemas
- Log errors with structured logging (logger.error())

### Anti-patterns to avoid
- Don't use `any` type in TypeScript
- Don't commit console.log statements
- Don't skip error handling

## Review Checklist

### Architecture
- [ ] Follows layered architecture (routes → services → models)
- [ ] No business logic in route handlers
- [ ] Proper separation of concerns

### Type Safety
- [ ] All functions have explicit return types
- [ ] No `any` types
- [ ] Input validated with Zod

### Testing
- [ ] Unit tests for business logic
- [ ] Integration tests for API routes
- [ ] Edge cases covered

### Security
- [ ] Input sanitized
- [ ] Authentication checked
- [ ] No secrets in code
```

<Tip>
The more specific your AGENTS.md, the better spec-loop's build and review phases will align with your standards.
</Tip>

## Verify Command Best Practices

### Fast feedback loops

<CodeGroup>
```bash Good: Incremental checks
VERIFY_COMMAND="npm run lint && npm run typecheck"
```

```bash Avoid: Slow full builds
VERIFY_COMMAND="npm run build"
```
</CodeGroup>

Verify command runs after every build. Keep it fast:
- Use incremental checking (tsc --noEmit, not full build)
- Lint only changed files if possible
- Skip slow tests (use TEST_COMMAND for those)

### Clear error output

```text
# Good: Clear failure messages
VERIFY_COMMAND="eslint . && tsc --noEmit"

# Bad: Silent failures
VERIFY_COMMAND="eslint . > /dev/null 2>&1"
```

Review phase needs clear errors to generate fix instructions.

### Exit codes matter

```text
# Ensure commands fail properly
VERIFY_COMMAND="npm run lint && npm run typecheck"

# Not: Always succeed
VERIFY_COMMAND="npm run lint || true"
```

spec-loop relies on exit codes to detect failures.

## Troubleshooting Configuration

### spec-loop can't find .speclooprc

**Symptom:** `No .speclooprc found. Run 'spec-loop init' first.`

**Solution:**
```bash
cd /path/to/your/project
spec-loop init
```

Run from project root, not subdirectory.

### Verify command fails every time

**Symptom:** Build phase always reports verify failure.

**Debug:**
```bash
# Test verify command manually
npm run lint && npm run typecheck  # or your command

# Check exit code
echo $?
```

Fix issues in your codebase or update `VERIFY_COMMAND` if wrong.

### Test command takes too long

**Symptom:** Each iteration takes minutes.

**Solutions:**
```bash
# Option 1: Use faster test subset
TEST_COMMAND="npm test -- --testPathPattern=src"

# Option 2: Skip slow integration tests
TEST_COMMAND="npm test -- --testPathIgnorePatterns=integration"

# Option 3: Empty test command, test manually
TEST_COMMAND=""
```

### Claude Code not found

**Symptom:** `Required command not found: claude`

**Solution:**
```bash
# Check if claude is installed
which claude

# If in custom location, configure
echo 'export CLAUDE_BIN="/usr/local/bin/claude"' >> ~/.bashrc
source ~/.bashrc

# Or in .speclooprc
CLAUDE_BIN="/usr/local/bin/claude"
```
